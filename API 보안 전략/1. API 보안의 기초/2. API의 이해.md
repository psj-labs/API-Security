# 2. API의 이해
---
# 1. HTTP 기본 구조

- **HTTP는 클라이언트와 서버가 데이터를 주고받는 통신 규칙입니다.**

---

## HTTP 요청 구조

HTTP 요청은 다음과 같은 구조로 구성됩니다.

```
Request Line
Headers
Body
```

### 예시

```
POST /login HTTP/1.1
Host: example.com
Content-Type: application/json

{"id":"admin","pw":"1234"}
```

---

## HTTP 응답 구조

```
Status Line
Headers
Body
```

### 예시

```
HTTP/1.1 200 OK
Content-Type: application/json

{"result":"success"}
```

---

## 취약점 포인트

- GET으로 상태 변경 → CSRF
- 입력값 검증 미흡 → SQL Injection
- Header 검증 부재 → 인증 우회
- 상태코드 오남용 → 정보 노출

---

# 2. URL 구조

- **URL은 서버 자원을 식별하는 주소 체계입니다.**

```
scheme://host:port/path?query
```

### 예시

```
https://example.com:443/users?id=1
```

### 구성 요소

| 요소 | 설명 |
|------|------|
| scheme | http / https |
| host | 서버 도메인 |
| port | 통신 포트 |
| path | 자원 위치 |
| query | 추가 파라미터 |

---

## 취약점 포인트

- IDOR
- SSRF
- Path Traversal
- Open Redirect

---

# 3. HTTP 메서드

- **메서드는 서버에게 어떤 동작을 수행할지 지시하는 명령어입니다.**

| 메서드 | 의미 |
|--------|------|
| GET | 조회 |
| POST | 생성 |
| PUT | 전체 수정 |
| PATCH | 일부 수정 |
| DELETE | 삭제 |

---

## 취약점 포인트

- GET으로 중요 기능 수행
- DELETE 권한 검증 누락
- PUT/PATCH 무분별 허용

---

# 4. 상태 코드

- **상태 코드는 요청 처리 결과를 나타내는 숫자 코드입니다.**

| 코드 | 의미 |
|------|------|
| 200 | 성공 |
| 201 | 생성 성공 |
| 301 | 리다이렉트 |
| 401 | 인증 필요 |
| 403 | 권한 없음 |
| 404 | 리소스 없음 |
| 500 | 서버 오류 |

---

# 5. REST API

- **REST는 자원을 URL로 표현하고 HTTP 메서드로 조작하는 설계 방식입니다.**

### 예시

```
GET /users/1
POST /users
DELETE /users/1
```

---

## REST 특징

- Stateless
- Cache 가능
- 자원 중심 설계

---

# 6. GraphQL

- **GraphQL은 클라이언트가 필요한 데이터만 요청하는 API 방식입니다.**

### 예시

```
{
  user(id:1){
    name
    email
  }
}
```

---

## 취약점 포인트

- 과도한 쿼리 요청 (DoS)
- 정보 과다 노출
- 복잡도 공격

---

# 7. 인증(Authentication)

- **인증은 사용자가 누구인지 확인하는 과정입니다.**

---

## 7-1 Basic 인증

```
Authorization: Basic base64(id:pw)
```

취약점:
- 평문 인코딩
- 재사용 가능
- MITM 공격 취약

---

## 7-2 API Key

```
x-api-key: abcdef123456
```

- 클라이언트 식별 목적
- 노출 시 서비스 남용 가능

---

## 7-3 HMAC 인증

- **HMAC은 요청에 서명을 추가하여 위변조를 방지하는 방식입니다.**

```
Signature = HMAC(secret, request)
```

- Replay Attack 방지 가능
- 서버와 클라이언트만 공유하는 비밀키 사용

---

# 8. OAuth 2.0

- **OAuth는 제3자 서비스에 비밀번호를 공유하지 않고 권한을 위임하는 인증 방식입니다.**

---

## 기본 흐름

1. 사용자 로그인
2. Authorization Code 발급
3. Access Token 발급
4. 리소스 접근

---

## 취약점 포인트

- Redirect URI 조작
- Authorization Code 탈취
- Access Token 유출

---

# 9. JWT (JSON Web Token)

- **JWT는 사용자 인증 정보를 담은 서명 기반 토큰입니다.**

---

## JWT 구조

```
Header.Payload.Signature
```

---

## Payload 예시

```
{
  "sub":"admin",
  "role":"admin",
  "exp":1712345678
}
```

---

## 취약점 포인트

- 만료시간 미설정
- 서명 검증 미흡
- LocalStorage 저장 (XSS 취약)
- 알고리즘 none 공격

---

# 10. 세션 & 쿠키

- **세션은 서버에 저장되는 사용자 상태 정보이며, 쿠키는 세션 ID를 저장하는 클라이언트 저장소입니다.**

```
Cookie: sessionid=abc123
```

---

## 취약점 포인트

- Weak Session ID
- HttpOnly 미설정
- Secure 미설정
- SameSite 미설정

---

# API 보안 핵심

- API 보안의 본질은 다음 두 가지입니다.

```
인증 (Authentication) → 누가 API를 사용하는가?
인가 (Authorization) → 그 사용자가 무엇을 할 수 있는가?
```

---

# API 취약점이 발생하는 근본 원인

- 인증 검증 부실
- 권한 검증 부실
- 입력값 검증 부재
- 토큰 관리 미흡
- 세션 관리 오류

---

# 정리

- HTTP 구조를 이해해야 API를 이해할 수 있습니다.
- 인증과 인가는 반드시 분리되어야 합니다.
- JWT는 서명 검증이 핵심입니다.
- OAuth는 권한 위임 프로토콜입니다.
- API 취약점의 대부분은 권한 검증 누락에서 발생합니다.
