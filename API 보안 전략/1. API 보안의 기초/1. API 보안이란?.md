# 1. API 보안이란?

## 개요

- 본 문서는 API 보안이 왜 중요한지, API 경제의 배경과 현실적인 문제점, 그리고 API 보안을 구성하는 기술적 요소들을 정리한 학습 문서입니다.
- 기존 작성 내용을 최대한 유지하면서 구조만 정리하고 필요한 부분만 보완하였습니다.

---

# 1. API 보안이 중요한 이유

- OWASP는 2019년 12월 최초의 **10대 API 보안 위협(API Security Top 10)**을 발표하였다.
- API 사용이 빠르게 확산되었고, API가 보호하는 데이터의 경제적 가치가 매우 높아졌다.
- 현재 API는 공격자들에게 가장 인기 있는 공격 대상이 되었다.

> API는 현대 서비스의 핵심 인터페이스이며 동시에 가장 노출된 공격 표면(Attack Surface)이다.

---

# 2. API 경제(API Economy)

## 개념

- API 경제란 비즈니스 또는 조직을 **플랫폼으로 전환하는 추동 요인**이다.

## 용도

- 서비스와 데이터를 소비자에게 유료로 제공
- 다른 제공업체의 서비스와 데이터를 활용하여 자신의 비즈니스를 개선
- 핵심 서비스 일부만 선택적으로 노출하여 기존 시스템을 파괴하지 않고 새로운 모델 수용

## 효과

- 더 많은 가치와 기능 제공
- 비용 절감
- 시장 출시 기간 단축(Time to Market 감소)

---

# 3. API 경제의 규모

- 전체 개발자의 90% 이상이 API 사용
- 전체 인터넷 트래픽의 84%가 API 요청/응답
- GitHub API 저장소 200만 개 이상
- APIM 산업 가치: 2023년 기준 51억 달러
- 통신 서비스 제공업체의 93%가 OpenAPI 명세 사용
- 91%의 조직이 API 보안 사고 경험

---

# 4. API 경제의 난제

## 1) 보유목록(재고) 관리 문제

- API는 구축과 배포가 쉬움
- 수명이 유한함
- 조직이 보유한 API 전체를 추적하기 어려움

### 결과

- 그림자 API (Shadow API)
- 좀비 API (Zombie API)

---

## 2) 거버넌스 문제

- API가 급증하면서 데이터 보호 요건 준수 어려움
- API 수명주기(개발 → 배포 → 운영 → 폐기) 전체 관리 어려움
- 가장 큰 난제는 보안

---

# 5. API는 왜 공격자에게 인기 있는가

1. API는 공개 접근이 가능할 가능성이 큼
2. 문서화가 잘 되어 있음
3. 자동화 공격 가능
4. 가치 있는 데이터를 직접 노출

---

## 개발자의 현실적 문제

- 꽃길 코딩(행복한 경로 코딩): 정상 흐름만 고려
- 예외 상황, 공격 시나리오 고려 부족
- 기술 부채가 많은 레거시 코드 유지
- 시스템 변경을 두려워함

## 1. 속도 제한 (Rate Limiting)

- 공용 API는 인터넷에 직접 노출됨
- 가장 단순한 공격 방법: 자동화 도구로 반복 요청

### 공격 예시

- DoS 공격
- 무차별 대입 공격
- 계정 탈취(ATO)

### 대응

- 일정 시간 동안 특정 IP 요청 횟수 초과 시
- HTTP 429 Too Many Requests 반환

---

## 2. 암호화

### 대칭 알고리즘

- 동일한 키 사용
- 속도 빠름
- 예: DES, AES, IDEA

### 비대칭 알고리즘

- 서로 다른 키 사용
- 예: RSA, ElGamal, DSS

---

### 암호화의 목적

1. 인증
2. 부인 방지
3. 기밀성
4. 무결성

---

## 3. 해시 / HMAC / 서명

### 해싱(Hash)

- 단방향 함수
- 데이터 블록 → 다이제스트 생성
- 무결성 검증에 사용

### HMAC

- 해시 + 대칭키
- 무결성 + 인증

### 디지털 서명

- 해시 + 비대칭키
- 무결성 + 인증 + 부인방지

---

### 다이제스트 비교표

| 목적 | 해시 | HMAC | 서명 |
|------|------|------|------|
| 무결성 | O | O | O |
| 인증 | X | O | O |
| 부인방지 | X | X | O |

---

## 4. 인코딩

- 인코딩은 표현 형식만 변경
- 암호화는 키를 이용해 데이터 보호

### 종류

- HTML 인코딩
- URL 인코딩
- ASCII, UTF-8
- Base64 (이진 데이터를 텍스트로 변환)

# 8. OpenAPI 명세

- API의 동작을 정의하는 사람이 읽기 쉬운 표준 규격

![01](/API%20보안%20전략/1.%20API%20보안의%20기초/imgs/01.png)


## 예시 정의서
1. OpenAPI 버전(openapi)
2. API 정보 메타데이터(info)
3. 호스트 URL을 포함한 서버 정보(servers)
```json
{
  "openapi": "3.0.0",
  "info": {
    "version": "1.0.0",
    "title": "Swagger Petstore",
    "license": {
      "name": "MIT"
    }
  },
  "servers": [
    {
      "url": "http://petstore.swagger.io/v1"
    }
  ]
}
```

---

## 종단점 구성 요소

1. 종단점 경로 이름(path)
2. HTTP 메서드
3. 요청 매개변수(parameters)
4. 상태 코드
5. 응답 형식(responses)

---

## 종단점 예시

```json
"paths": {
  "/pets": {
    "get": {
      "summary": "List all pets",
      "operationId": "listPets",
      "parameters": [
        {
          "name": "limit",
          "in": "query",
          "description": "Maximum items (max 100)",
          "required": false,
          "schema": {
            "type": "integer",
            "format": "int32"
          }
        }
      ],
      "responses": {
        "200": {
          "description": "A paged array of pets",
          "headers": {
            "x-next": {
              "description": "Next page",
              "schema": {
                "type": "string"
              }
            }
          }
        }
      }
    }
  }
}
```

---

## OpenAPI 명세 활용 기능

1. 정의서 검증 및 린팅 → 문법/표준 오류 검사  
2. 데이터 검증 → 요청/응답 스키마 검증  
3. 문서 생성 → 자동 API 문서화  
4. 코드 생성 → 서버/클라이언트 코드 자동 생성  
5. GUI 편집기 → 시각적 설계 도구  
6. 모의 서버(Mock Server) → 백엔드 없이 테스트  
7. 보안 분석 → 잘못된 설정 및 취약점 점검

# 10. APIM 포털과 게이트웨이

## API 게이트웨이

- API 트래픽 단일 진입점
- 트래픽 변환 및 전달
- 내부 위협 방어는 제한적

## APIM 포털

- API 카탈로그 관리
- 버전 관리
- 수명주기 관리
- 사용자 경험 통제

→ 둘 다 핵심 요소지만 전체 보안 전략을 대체하지는 못한다.

# 11. API 오남용 사례

- 허용된 행위자만 접근 허용
- OpenAPI 계약을 철저히 작성해야 효과적

# 정리

- API는 비즈니스 핵심 자산이다.
- API는 가장 노출된 공격 표면이다.
- 속도 제한, 암호화, 무결성 검증, 계약 기반 통제가 필수이다.
- OpenAPI 명세는 보안과 거버넌스의 핵심 도구이다.
- API 보안은 기술 + 거버넌스 + 운영을 모두 포함해야 한다.
