# OWASP API 보안 취약점 정리 (OWASP API Top 10 - 2019)

- 최근 대부분의 서비스가 웹 애플리케이션뿐 아니라 모바일 애플리케이션, SPA, IoT 등 다양한 환경에서   
   **API 기반으로 데이터를 교환**하기 때문에 API 보안의 중요성이 매우 높아졌습니다.

### API는 일반 웹 페이지와 달리 **데이터 중심 구조**이기 때문에 다음과 같은 문제가 발생하기 쉽습니다.

- 권한 검증 오류
- 인증 처리 취약점
- 과도한 데이터 노출
- 입력값 검증 부족
- 요청 제한 미구현

- 이러한 문제를 체계적으로 정리한 것이 **OWASP API Top 10**입니다.

---

# API1:2019 Broken Object Level Authorization (BOLA)

## 개념

- Broken Object Level Authorization은 **객체 수준 권한 검증이 제대로 이루어지지 않아 다른 사용자의 데이터에 접근할 수 있는 취약점**입니다.
- API에서는 데이터 객체를 식별하기 위해 보통 **ID 값**을 사용합니다.

### 예시
```
GET /api/accounts/123
```
- 이 요청은 ID가 123인 계좌 정보를 조회하는 요청입니다.
- 하지만 서버가 해당 객체에 대한 **접근 권한을 확인하지 않는 경우**, 공격자는 객체 ID 값을 변경하여 다른 사용자의 데이터를 조회할 수 있습니다.

---

## 취약점 발생 과정

1. 사용자가 자신의 데이터를 조회합니다.
```
GET /api/accounts/123
```
2. 공격자가 요청을 수정합니다.
```
GET /api/accounts/124
```
3. 서버가 권한 검증을 수행하지 않으면 다른 사용자의 데이터가 반환됩니다.

---

## 대응 방법

- 모든 객체 접근 요청에 대해 권한 검증 수행
- 객체 소유자 확인
- 접근 제어 로직 중앙화

---

# API2:2019 Broken Authentication

## 개념

- Broken Authentication은 **인증 로직이 제대로 구현되지 않아 공격자가 다른 사용자로 인증될 수 있는 취약점**입니다.

### API 환경에서는 다음과 같은 인증 방식이 많이 사용됩니다.
- JWT
- OAuth
- API Key
- Session 기반 인증

#### 이러한 인증 과정이 제대로 구현되지 않으면 인증 우회가 발생할 수 있습니다.

---

## 주요 원인

- 토큰 검증 미흡
- 토큰 만료 처리 미흡
- 약한 인증 정책
- 인증 로직 오류

---

## 대응 방법

- 강력한 인증 정책 적용
- 토큰 검증 강화
- MFA 적용

---

# API3:2019 Excessive Data Exposure

## 개념

- Excessive Data Exposure는 **API가 필요 이상의 데이터를 반환하는 취약점**입니다.

### 예시 응답
```json
{
 id: 10,
 username: "user",
 email: "user@email.com",
 password_hash: "...",
 is_admin: false
}
```
- 클라이언트는 username과 email만 필요하지만 서버는 **모든 데이터를 반환**하고 있습니다.
- 이 경우 공격자는 민감한 정보까지 확인할 수 있습니다.

---

## 대응 방법

- 최소 데이터 원칙 적용
- 서버 측 데이터 필터링
- 민감 정보 제거

---

# API4:2019 Lack of Resources & Rate Limiting

## 개념

- 이 취약점은 **API 요청 횟수나 시스템 자원 사용에 제한이 없는 경우 발생합니다.**

### 공격자는 다음과 같은 공격을 수행할 수 있습니다.

- 무차별 대입 공격
- API 크롤링
- 서비스 거부 공격

---

## 대응 방법

- Rate Limiting 적용
- 사용자별 요청 제한
- IP 기반 제한

---

# API5:2019 Broken Function Level Authorization

## 개념

- Broken Function Level Authorization은 **사용자의 권한 수준에 따라 접근이 제한되어야 하는 기능에 대해 권한 검증이 제대로 이루어지지 않는 취약점**입니다.
- API에서는 일반적으로 **관리자 기능과 일반 사용자 기능이 구분**되어 있습니다.

### 예시

- 관리자 전용 API
```
GET /api/admin/users
```
- 이 API는 관리자만 접근해야 합니다.
- 하지만 권한 검증이 제대로 구현되지 않으면 일반 사용자도 해당 API에 접근할 수 있습니다.

---

## 공격 예시

- 일반 사용자 요청
```
GET /api/user/profile
```
- 공격자가 관리자 기능을 호출
```
GET /api/admin/users
```
- 서버가 권한 검증을 하지 않는 경우 **관리자 기능이 실행될 수 있습니다.**

---

## 대응 방법

- 역할 기반 접근 제어(RBAC) 적용
- 모든 기능에 대해 권한 검증 수행
- 관리자 기능 분리

---

# API6:2019 Mass Assignment

## 개념

- Mass Assignment는 **클라이언트가 전달한 데이터를 그대로 객체에 매핑하는 과정에서 발생하는 취약점**입니다.

#### 예를 들어 사용자 정보 수정 API가 있다고 가정합니다.
```
PUT /api/user/update
```
#### 요청 데이터
```json
{
 username: "user1",
 email: "user1@email.com"
}
```
- 서버가 이 데이터를 그대로 사용자 객체에 매핑한다면 정상적으로 동작합니다.
- 하지만 공격자가 다음과 같은 데이터를 추가할 수 있습니다.
```json
{
 username: "user1",
 email: "user1@email.com",
 is_admin: true
}
```
- 서버가 이를 검증하지 않으면 **일반 사용자가 관리자 권한을 획득할 수 있습니다.**

---

## 대응 방법

- 허용된 필드만 업데이트
- 입력 필드 화이트리스트 적용
- 민감 필드 보호

---

# API7:2019 Security Misconfiguration

## 개념

- Security Misconfiguration은 **보안 설정이 잘못되어 발생하는 취약점**입니다.

#### 대표적인 예시

- 디버그 모드 활성화
- 기본 계정 사용
- CORS 설정 오류
- 불필요한 서비스 활성화

---

# API8:2019 Injection

## 개념

- Injection 취약점은 **사용자 입력이 검증되지 않고 시스템 명령이나 쿼리에 포함되는 경우 발생합니다.**

### 대표적인 예

- SQL Injection
- NoSQL Injection
- Command Injection

---

# API9:2019 Improper Asset Management

## 개념

- Improper Asset Management는 **API 자산을 제대로 관리하지 않아 발생하는 취약점**입니다.

### 예시

- 오래된 API 버전
- 문서화되지 않은 API
- 테스트 API 노출

---

# API10:2019 Insufficient Logging & Monitoring

## 개념

- 이 취약점은 **로그 기록 및 모니터링이 부족하여 공격을 탐지하지 못하는 상황**을 의미합니다.
#### 공격자가 다음과 같은 행동을 수행해도 탐지되지 않을 수 있습니다.

- 계정 탈취
- 데이터 유출
- API 오용

---

# 정리

- OWASP API Top 10은 API 환경에서 가장 중요한 보안 취약점 목록입니다.

#### 특히 다음 취약점은 실제 서비스에서 매우 자주 발생합니다.

- Broken Object Level Authorization
- Broken Authentication
- Excessive Data Exposure
- Broken Function Level Authorization
- Mass Assignment

#### API 보안을 설계할 때 가장 중요한 원칙은 다음과 같습니다.

- 서버에서 권한 검증 수행
- 최소 데이터 노출
- 입력값 검증
- 요청 제한 적용
- 로그 및 모니터링 강화

### **이러한 보안 원칙을 적용하면 대부분의 API 취약점을 예방할 수 있습니다.**
